    아바타 애니메이션 기법 연구 논의¶

    날짜: 2026-03-08 상태: 검토 중 (방향 미확정)

    1. 배경¶

    게임에서 사용할 아바타 캐릭터 애니메이션을 가볍고 효율적으로 제작/구현하는 방법을 연구 중.

    현재 기술 환경¶

    • Unity + NGUI (향후 UGUI 전환 예정)
    • 2D 게임, Built-in 렌더 파이프라인
    • Spine 사용 중 (FFD 적극 활용: 머리카락, 눈, 얼굴, 의상)

    게임 내 아바타 스펙¶

    • 사이즈: 240x260 (가슴까지 프로필)
    • 동시 표시: 5명
    • 애니메이션: 승리 모션 1종
    • 평소 상태: 정지 이미지 1장

    2. 검토한 접근 방식¶

    접근 1: AI 영상 생성 → 알파 추출 → WebM¶

    원화를 AI 영상으로 변환 후 배경 제거하여 WebM으로 게임에 사용.

    연구 내용: - differential matting 기법 구현 완료 (초록+파란 배경 2장 → 수학적 알파 추출) - 정지 이미지에서는 완벽하게 동작

    문제점: - AI 영상 생성 시 동일 영상을 배경만 바꿔서 2개 생성 불가능 - 단일 배경 크로마키는 반투명 영역(머리카락 등) 품질 한계 - 프레임마다 AI의 미세한 떨림 → 알파 경계 불안정 - WebM 고품질 알파 포함 시 파일 크기 부담

    결론: 현재 기술로는 안정적이지 않음

    접근 2: Spine 데이터 → Unity 네이티브 프리팹 변환¶

    Spine 런타임 제거, Spine 데이터를 Unity 네이티브(Transform + SpriteRenderer/UI Image)로 변환.

    변환 가능: - 본 트랜스폼 (이동/회전/스케일) - 스프라이트 교체, 컬러/투명도, 드로우 오더, 스킨, 이벤트

    포기해야 하는 것: - 메시 디포메이션 (FFD) - IK 제약조건 - 패스/트랜스폼 제약조건 - 클리핑(마스크) - 메시 기반 어태치먼트 - 비선형 커브 보간 (미세 차이 가능)

    문제점: - 현재 프로젝트에서 FFD를 머리카락, 눈, 얼굴, 의상에 많이 사용 중 - FFD 포기 시 캐릭터 퀄리티가 크게 하락

    결론: FFD 의존도가 높아서 단순 변환은 어려움

    접근 3: Spine 제작 → 스프라이트 시트 렌더링 추출¶

    디자이너는 Spine으로 자유롭게 작업 (FFD, 마스크, IK 모두 사용), 빌드 시 프레임별 렌더링하여 스프라이트 시트로 추출.

    장점: - 디자이너 작업 방식 변경 없음 - Spine 런타임 완전 제거 - 마스크/셰이더/유지보수 문제 원천 제거 - NGUI/UGUI 어디든 Image 컴포넌트로 재생 - 드로우콜 캐릭터당 1개

    단점: - 텍스처 메모리 증가 (프레임 수 x 이미지 크기)

    메모리 계산 (240x260 기준): - 최적화 전: 30프레임 x 243KB = 약 7.3MB/캐릭터 - 최적화 후 (10fps + ETC2/ASTC 압축 + 짧은 루프): 약 300KB/캐릭터 - 5명 = 약 1.5MB → 모바일에서도 가능한 수준

    추가 최적화 가능: - 승리 시에만 시트 로드, 끝나면 해제 - 평소 정지 이미지는 1장만 유지

    결론: 현재 스펙(240x260, 승리 1종)이면 현실적으로 가능

    3. Spine + Unity 실무 문제점 (개발팀 피드백)¶

    셰이더 문제¶

    • Spine 자체 셰이더 ↔ Unity 렌더 파이프라인 충돌
    • 커스텀 이펙트 추가 시 Spine 셰이더 직접 수정 필요 (높은 난이도)

    머티리얼 문제¶

    • 파츠별 블렌딩 모드 상이 → 머티리얼 분리 → 드로우콜 증가
    • 아틀라스 관리 복잡

    마스크 문제 (가장 심각)¶

    • Spine 클리핑 = 스텐실 버퍼 사용
    • 마스크마다 드로우콜 추가 발생
    • 마스크 중첩 시 프레임 드랍 (모바일에서 특히 심각)
    • 렌더링 순서 꼬임 가능

    유지보수 문제¶

    • Spine 파일(디자이너) + 코드(개발자) 양쪽 이해 필요
    • Spine Runtime 버전 업데이트 호환성 문제
    • 복잡한 캐릭터일수록 구조 비대화

    NGUI/UGUI 연동 문제¶

    • Spine은 MeshRenderer 기반, UI는 Canvas 기반
    • RenderTexture 우회 또는 SkeletonGraphic 필요 (제약 많고 성능 이슈)
    • NGUI → UGUI 전환 시 Spine 연동 코드 전면 재작업

    4. 핵심 방향 전환: AI 활용 필요성¶

    Spine의 근본 문제 - 제작 속도¶

    • 기술적 이슈(마스크/셰이더/유지보수)뿐 아니라 제작 속도가 핵심 문제
    • 캐릭터 1개 승리 모션 = 파츠 분리 + 리깅 + 애니메이션 + QA → 공수 큼
    • 다양한 승리 모션을 원하면 작업량 x배로 증가
    • AI는 원화 1장 + 프롬프트만 바꾸면 다양한 모션을 빠르게 생성 가능

    AI 활용 목표¶

    다양한 승리 모션을 빠르게 대량 생산하고 싶다 → Spine으로는 불가능 → AI로 해결

    5. 배경 분리 없이 영상 재생 방식 검토¶

    알파 추출 문제를 우회하여, 배경을 분리하지 않고 AI가 생성한 배경 그대로 영상을 재생하는 방식 검토.

    접근 4: WebM 영상 재생 (배경 포함)¶

    장점: 알파 추출 불필요, 용량 극소 (50~100KB)

    문제점: - iOS에서 WebM 미지원 (Unity VideoPlayer는 플랫폼 네이티브 디코더 사용) - VideoPlayer 1개당 별도 디코더 스레드 → 5명 동시 재생 시 모바일 부담 - UI Canvas 위에 올리려면 RawImage + RenderTexture 필요 (5개 = 추가 메모리) - AI 배경색과 게임 UI 배경의 정확한 매칭 필요

    결론: iOS 미지원이 치명적

    접근 5: 스프라이트 시트 (배경 포함, 알파 없음)¶

    배경 분리 없이 프레임 이미지를 스프라이트 시트로 구성.

    메모리 계산 (240x260, 10fps, 5초 = 50프레임): - 스프라이트 시트: 약 2048x2048 한 장에 배치 가능 - ETC2 압축: 약 2.7MB/캐릭터 - 5명 동시: 약 13.5MB - 캐릭터 종류 증가 시 더 심각

    결론: 텍스처 메모리 과다 → 비현실적

    접근 6: MP4 (H.264) 영상 재생 (유력 후보)¶

    배경 분리 없이, WebM 대신 MP4 포맷 사용.

    항목 WebM MP4 (H.264)
    iOS 미지원 지원
    Android 지원 지원
    알파 채널 VP9만 가능 불가 (배경 포함이니 불필요)
    240x260 5초 용량 ~80KB ~50KB
    디코더 소프트웨어 하드웨어 가속

    장점: - iOS/Android 모두 지원 - 하드웨어 디코딩 → 소프트웨어보다 훨씬 가벼움 - 용량 극소 (50~100KB/캐릭터) - 알파 추출 불필요 - AI 영상을 거의 그대로 사용 가능

    확인 필요: - 동시 5개 재생이 모바일 하드웨어 디코더에서 안정적인지 - UI Canvas 위에 RenderTexture로 올릴 때 성능 - AI 배경색과 게임 UI 배경 매칭 품질

    결론: 프로토타입 제작 및 실기기 테스트 필요

    6. 추가 검토 및 탈락¶

    MP4 색감 문제¶

    • H.264는 YUV 4:2:0 색공간 사용 → 색 정보가 원본 대비 1/4로 손실
    • 선명한 색 경계 뭉개짐, 채도 저하, 빨강/파랑 번짐
    • 2D 셀룩/일러스트 스타일에서 특히 심각 (실사보다 더 티남)
    • 240x260 작은 사이즈에서 더 눈에 띌 수 있음

    결론: MP4도 2D 일러스트 아바타에는 부적합

    APNG¶

    • 프레임마다 PNG 통째로 담는 구조 → 스프라이트 시트와 비슷한 용량
    • Unity 네이티브 지원 없음 (별도 라이브러리 필요)

    결론: 용량 문제 해결 안 됨 → 탈락

    7. 전체 딜레마 요약¶

    방식 핵심 문제
    WebM 영상 재생 iOS 미지원, 동시 5개 디코더 부담
    MP4 영상 재생 YUV 4:2:0 색감 손실 (2D 일러스트에 치명적)
    APNG 용량 무거움, Unity 미지원
    스프라이트 시트 (풀프레임) 텍스처 메모리 과다 (5초 분량)
    Spine 마스크/셰이더/유지보수/제작속도
    AI 알파 추출 (영상) 품질 불안정

    완벽한 단일 방법은 없음 → 용도별 분리 전략으로 결론

    8. 최종 방향: 용도별 분리 전략¶

    핵심 원칙¶

    모든 곳에 하나의 방식을 쓰려 하지 말고, 용도에 맞는 최적의 방식을 선택한다.

    용도별 적용¶

    용도 방식 이유
    전투/승리 등 핵심 연출 Spine 퀄리티가 중요, FFD 필요
    대기실 안내 아바타 Unity 하이어라키 단순 반복 동작, 가벼움
    튜토리얼 캐릭터 Unity 하이어라키 일회성, 유지보수 쉬움
    로비/상점 NPC 등 Unity 하이어라키 숨쉬기/끄덕임 정도면 충분

    Unity 하이어라키 방식 상세¶

    • 파츠 분리 → Transform 계층 → AnimationClip
    • FFD 없이 본 회전/이동/스케일 + 파츠 교체만
    • 메모리 최소 (파츠 이미지 몇 장)
    • Spine 런타임 불필요 → 셰이더/마스크 문제 없음
    • NGUI/UGUI 어디든 자연스럽게 동작
    • 개발자가 완전히 제어 가능

    AI 활용 포인트¶

    • 하이어라키 캐릭터의 파츠 이미지 생성에 AI 활용
    • 원화 1장에서 표정 변형, 포즈 변형 등 생성
    • 정지 이미지 기반이므로 differential matting으로 깔끔한 알파 추출 가능

    이 전략의 이점¶

    • Spine 사용 범위 축소 → 유지보수 부담 감소
    • 하이어라키 부분은 개발자가 자유롭게 제어
    • NGUI → UGUI 전환 시 하이어라키 부분은 수월
    • AI 활용으로 파츠 제작 속도 향상

    9. AI 눈 애니메이션 자동화 파이프라인¶

    개요¶

    AI 영상에서 눈 영역만 추출하여 NGUI 스프라이트 시트 + Unity AnimationClip을 자동 생성하는 도구. 수동 작업을 최소화하여 대기실/튜토리얼 등 보조 캐릭터의 눈 애니메이션을 빠르게 제작.

    파이프라인 흐름¶

    AI 눈 영상 생성 (수동)
    ↓
[eye_anim_pipeline.py --face] 자동 처리 (2-pass):
    ├→ Pass 1: MediaPipe Face Mesh로 양쪽 눈 랜드마크 검출 (478점)
    ├→ 가우시안 스무딩 + 아핀 정규화 (스케일/회전/위치 보정)
    ├→ Pass 2: 정규화된 좌표로 크롭 (jitter 0px)
    ├→ 중복 프레임 자동 제거 (유사도 기반)
    ├→ 스프라이트 시트 생성 (POT, RGBA)
    ├→ Unity Legacy AnimationClip (.anim) 생성
    └→ C# 헬퍼 스크립트 생성 (frameIndex → spriteName 매핑)
    ↓
Unity에서 Import → NGUI Atlas 등록 → 바로 사용

    아핀 정규화 (Affine Normalization)¶

    기준 프레임의 양쪽 눈 위치를 기준으로 매 프레임을 아핀 변환: - 이동 보정: 눈 중심을 항상 같은 위치에 배치 - 스케일 보정: 줌인/아웃 시 눈 간 거리를 일정하게 유지 - 회전 보정: 고개 기울임에 따른 눈 각도 일정하게 유지 - 결과: 프레임 간 떨림 0px, 표정 변화만 자연스럽게 캡처

    눈 추적 방식 비교¶

    방식 jitter 줌/회전 대응 비고
    고정 크롭 (--crop) 0px (고정) 불가 정지 카메라용
    템플릿 매칭 (--track) 드리프트 불가 레거시
    Face Mesh 1-pass avg 1.18px 대응 약간 흔들림
    Face Mesh + 아핀 (--face) 0px 완벽 대응 권장

    테스트 결과 (grok_sample.mp4 - 카메라 줌인+회전 영상): - 템플릿 매칭: 후반부 코/입/머리카락 포착 (실패) - Face Mesh + 아핀: 전 프레임 눈 정확 추적, 떨림 없음 (성공)

    사용법 (권장: Face Mesh 모드)¶

    python eye_anim_pipeline.py \
  --video eye_blink.mp4 \
  --face \
  --eye-size 200,80 \
  --name char01_eye \
  --fps 10 \
  --output ./output/eye_anim \
  --dedup 0.95

    사용법 (고정 크롭 - 정지 카메라용)¶

    python eye_anim_pipeline.py \
  --video eye_blink.mp4 \
  --crop 100,50,60,30 \
  --name char01_eye \
  --fps 10 \
  --output ./output/eye_anim

    출력물¶

    • {name}_sheet.png : 스프라이트 시트 (NGUI Atlas에 등록)
    • {name}_sprites.json : 스프라이트 좌표 메타데이터
    • {name}.anim : Unity Legacy AnimationClip (float curve로 frameIndex 제어)
    • {ClassName}.cs : 헬퍼 스크립트 (frameIndex → UISprite.spriteName 매핑)

    Unity 적용 순서¶

    1. _sheet.png를 NGUI Atlas에 등록 (좌표는 _sprites.json 참조)
    2. .anim 파일을 Unity에 Import
    3. .cs 헬퍼 스크립트를 눈 오브젝트에 부착
    4. targetSprite에 UISprite 연결
    5. Animation 컴포넌트에 .anim 등록 후 재생

    기술 특징¶

    • MediaPipe Face Mesh + 아핀 정규화: 2-pass로 눈 위치 완전 안정화 (0px jitter)
    • Legacy Animation 방식 (프로젝트 기존 방식과 동일)
    • 중복 프레임 자동 제거 → 아틀라스 용량 절약
    • POT 텍스처 → 모바일 호환
    • float curve + 헬퍼 스크립트 방식 → NGUI UISprite와 안정적 연동
    • 양쪽 눈/왼쪽만/오른쪽만 선택 가능 (--eye-which)
    • 궤적 스무딩 강도 조절 가능 (--smooth, 기본 5)
    • 모델: models/face_landmarker.task (3.6MB, Google 제공)
